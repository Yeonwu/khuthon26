# 프로젝트 개요

만들려는 서비스는 **음원 일부 또는 레퍼런스 샘플을 업로드하면, 그 소리와 유사한 국악/전통 타악기 샘플을 찾아주는 검색·추천 서비스**다.

핵심 사용자는 음악 프로듀서다. 프로듀서는 일반적인 드럼 킥, 퍼커션, 루프 대신 전통 타악기 소리를 사용하고 싶지만, 국악 샘플을 직접 탐색하기 어렵다. 이 서비스는 업로드한 레퍼런스 샘플과 **리듬, 어택, 질감이 유사한 전통 타악기 샘플**을 찾아준다.

최종적으로는 “곡에 어울리는 샘플 추천”이 이상적이지만, “어울리는”은 주관적이고 평가가 어렵기 때문에 MVP에서는 **“유사한 샘플 검색”**으로 정의했다.

---

# 문제 정의

아티스트가 원하는 것은 단순히 “국악 샘플”이 아니다. 실제로는 다음과 같은 식의 요구다.

> “이 킥이랑 비슷한데 더 둔탁한 북소리”
> “이 퍼커션 루프와 비슷한 리듬의 전통 타악기”
> “이 드럼 패턴을 전통 북 계열 소리로 대체하고 싶다”
> “너무 전통적으로 들리진 않지만, 국악기 특유의 날것 같은 질감이 필요하다”

따라서 서비스의 가치는 **국악 아카이브를 보여주는 것**이 아니라, 프로듀서가 쓰는 언어와 작업 흐름에 맞춰 **전통 타악기 샘플을 검색 가능한 형태로 번역하는 것**에 있다.

---

# 핵심 사용 시나리오

프로듀서는 곡에 일반 드럼 킥이나 퍼커션 대신 **전통 북 계열의 타격감**을 넣고 싶다.

하지만 “북소리 같은데 현재 드럼 패턴과 비슷한 리듬·어택·질감을 가진 샘플”을 직접 찾기 어렵다.

그래서 프로듀서는 자신이 만든 드럼 루프, 킥 패턴, 퍼커션 루프, 또는 레퍼런스 샘플을 서비스에 업로드한다.

서비스는 해당 음원의 특성을 분석하고, 이와 유사한 국악 타악기 샘플을 추천한다.

추천 결과는 원본 국악 샘플로 반환된다. 사용자는 이를 DAW에서 기존 드럼을 대체하거나 레이어링하는 데 사용할 수 있다.

한 줄 요약:

> 사용자가 기존 드럼 루프나 레퍼런스 샘플을 업로드하면, 리듬·어택·질감이 유사한 전통 타악기 샘플을 찾아주는 서비스.

---

# “어울리는”과 “유사한”의 관계

처음에는 “업로드한 음원에 어울리는 샘플 추천”을 목표로 잡았다.

하지만 “어울리는”은 주관적이다. 사람마다 음악적 취향, 장르, 믹싱 방식, 편곡 의도가 다르기 때문에 알고리즘적으로 명확히 정의하기 어렵다.

그래서 MVP에서는 “어울리는”을 직접 예측하기보다, **업로드한 샘플과 유사한 샘플을 찾는 것**으로 범위를 좁혔다.

다만 최종 서비스 가치는 단순 유사성에만 있으면 약하다.

논의된 개념은 다음과 같다.

```text
어울리는 샘플 = 유사성 + 보완성 + 사용 가능성
```

MVP에서는 이 중 **유사성**만 먼저 구현한다.

즉, 서비스 메시지는 이렇게 잡는 것이 적절하다고 봤다.

> 업로드한 음원의 리듬·타격감·음색 질감과 유사한 전통 타악기 샘플을 찾아, 창작자가 바로 변주하거나 레이어링할 수 있게 추천한다.

---

# 샘플 탐색 방식에 대한 논의

일반적으로 아티스트가 샘플을 찾는 방식은 다음과 같다.

1. 레퍼런스 기반 검색
2. 장르·악기·질감 키워드 검색
3. Splice, Loopcloud, Tracklib, YouTube, Freesound, Bandcamp, SoundCloud 같은 플랫폼 탐색
4. 기존 음원에서 직접 잘라 사용
5. 질감, 어택, 잔향, BPM, Key, 루프성, 저작권 등을 기준으로 선택

중요한 판단은 이것이다.

아티스트는 “국악” 자체를 찾는 게 아니라 **자신의 트랙에서 특정 역할을 해줄 소리**를 찾는다.

예를 들면:

* 킥 대체
* 드럼 레이어링
* 퍼커션 루프
* 브레이크
* 드론
* 질감
* 인트로 소스
* 불안정한 텍스처

따라서 서비스 검색축도 “국악 장르” 중심보다 다음과 같은 축이 더 적절하다.

* 악기명: 북, 장구, 꽹과리, 징, 소리북, 좌고 등
* 소리 성질: 둔탁한, 날카로운, 거친, 건조한, 금속성, 낮은, 찢어지는
* 음악적 용도: 킥 대체, 하이햇 대체, 레이어링, 텍스처, 브레이크
* 리듬 특성: 반복적, 빠른, 느린, 밀도 높은, 엇박
* 분위기: 불안한, 원초적인, 의식적인, 미니멀한, 어두운

# 분석 기준: 리듬·어택·질감

처음에는 타악기 유사도를 분석할 때 다음 세 가지 축을 고려했다.

## 1. 리듬

타악기에서는 언제 소리가 나는지가 중요하다.
따라서 onset pattern, onset interval, 강세 위치, BPM, tempogram, beat histogram 등이 후보로 논의됐다.

## 2. 어택

어택은 타격 순간의 성질이다.

예:

* 얼마나 빠르게 튀어나오는가
* 얼마나 강한가
* decay가 짧은가 긴가
* 킥처럼 단단한가, 북처럼 퍼지는가

RMS envelope, attack slope, peak RMS, decay time 등을 사용할 수 있다고 봤다.

## 3. 질감

질감은 음색이다.

예:

* 둔탁한가
* 밝은가
* 거친가
* 금속성인가
* 저역 중심인가
* 긁히는 느낌이 있는가

MFCC, spectral centroid, spectral bandwidth, spectral flatness, spectral rolloff 등이 후보였다.

초기에는 다음과 같은 신호처리 기반 하이브리드 구조를 검토했다.

```text
onset pattern + MFCC + spectral centroid + RMS envelope
```

이 방식은 설명 가능성이 높지만, 직접 feature engineering을 해야 하고 MVP 구현 부담이 있다.

# MERT 사용 논의

이후 MERT를 활용해 한 번에 오디오 임베딩을 추출하는 방식을 검토했다.

MERT는 음악 오디오를 위한 self-supervised acoustic model이고, 음악적 특징을 임베딩으로 뽑을 수 있다.

논의된 판단은 다음과 같다.

## MERT만 사용하는 장점

* 구현이 단순하다.
* MFCC, centroid, RMS 등을 직접 설계하지 않아도 된다.
* 음색, 악기감, 질감, 분위기 유사도는 비교적 잘 잡을 수 있다.
* 해커톤 MVP에 적합하다.
* 전통 타악기 루프 유사 검색에서는 충분히 설득력 있다.

## MERT만 사용하는 단점

* 리듬 패턴 유사도를 정확히 보장하지는 못한다.
* 음색은 비슷하지만 리듬이 다른 샘플이 나올 수 있다.
* 설명 가능성이 낮다.
* 긴 루프와 짧은 루프 비교가 불안정할 수 있다.

최종 판단은 다음과 같다.

> 해커톤 MVP에서는 MERT-only로 시작해도 된다.
> 다만 실제 프로덕트로 고도화할 때는 BPM, onset density, onset pattern 같은 리듬 보정을 추가하는 것이 좋다.

발표나 설명에서는 이렇게 표현하는 것이 안전하다고 봤다.

> 초기 버전은 MERT 기반 오디오 임베딩으로 유사 샘플을 검색하고, 이후 리듬 정확도를 높이기 위해 onset/BPM 기반 재정렬을 추가할 수 있다.

# 데이터 조건

중요한 데이터 조건도 논의됐다.

국악 샘플 데이터는 다음과 같은 특징을 가진다.

* 한 샘플 안에 한 악기만 있음
* 리듬은 반복 구조임
* one-shot은 제외하고 loop만 포함
* 전체 길이는 1초~20초
* 중앙값은 약 12초
* 평균값은 약 10초

이 조건 때문에 긴 루프 안에 여러 악기가 섞여 평균화되는 문제는 줄어든다.

즉, 한 샘플이 “북 + 장구 + 꽹과리 + 쉼”처럼 복합적으로 구성된 것이 아니라, 대부분 **단일 악기의 반복 리듬 루프**라고 전제했다.

이 조건에서는 **전체 루프 단위 MERT 임베딩**도 어느 정도 합리적이라고 판단했다.

다만 길이 차이는 여전히 문제가 된다.

---

# 루프 길이 문제

처음에는 모든 루프를 4초 고정 segment로 자르는 방식을 검토했다.

하지만 “왜 4초인가?”라는 질문이 나왔고, 4초는 서양음악 기준에서 120 BPM 4/4박자의 2마디 정도에 해당하는 임의적 타협점이라는 점을 확인했다.

국악에는 서양식 “마디” 개념을 그대로 적용하기 어렵다.
국악은 마디보다 **장단 cycle** 중심으로 보는 것이 적절하다.

예:

* 세마치
* 굿거리
* 자진모리
* 휘모리
* 진양조

따라서 국악 타악기 샘플을 단순히 “4초 = 1마디”로 가정하는 것은 부정확하다고 봤다.

이상적으로는 다음이 맞다.

```text
고정 초 단위 segment보다 장단 cycle 단위 segment가 더 적절하다.
```

하지만 자동 장단 검출은 구현 난도가 높다.
그래서 MVP에서는 초 단위 정규화 또는 MERT context 기준을 사용할 수밖에 없다고 봤다.

---

# 길이를 자르지 않고 비교하는 경우의 부작용

루프 전체를 그대로 MERT에 넣고 벡터 하나로 비교하는 경우 다음 문제가 논의됐다.

## 1. 긴 루프 평균화

긴 루프 안에 변화가 있으면 전체 벡터가 평균화된다.
특정 구간이 query와 매우 비슷해도 전체 평균에서는 점수가 낮아질 수 있다.

## 2. 짧은 루프와 긴 루프 비교 불공정

3초 query와 20초 DB sample을 직접 비교하면, 20초 안의 일부가 매우 비슷해도 전체 벡터에서는 희석될 수 있다.

## 3. 리듬 구조가 흐려짐

반복 주기, 강세, 필인, 쉼 등이 하나의 벡터로 섞인다.

## 4. 앞뒤 무음·잔향 영향

앞뒤 무음, 잔향, 준비음, 환경음 등이 임베딩에 영향을 줄 수 있다.

## 5. 부분적으로 좋은 샘플을 놓칠 수 있음

프로듀서가 실제로 쓰고 싶은 것은 전체 20초가 아니라 그 안의 4~5초짜리 좋은 구간일 수 있다.

다만 현재 데이터가 “한 악기, 반복 리듬” 중심이기 때문에 이 문제는 복합 음원보다 덜 심각하다고 봤다.

# 최종적으로 논의된 길이 처리 방향

최종적으로 다음 방향이 제안됐다.

## 4초 미만 샘플

4초보다 짧은 루프는 반복해서 4초 이상으로 맞춘다.

예:

```text
1초 루프 → 4번 반복
2초 루프 → 2번 반복
3초 루프 → 반복 후 4초 이상으로 맞춤
```

무음 padding은 피하는 것이 좋다고 봤다.

이유:

* 에너지가 낮아진다.
* 리듬 밀도가 왜곡된다.
* MERT가 “쉼이 많은 루프”처럼 인식할 수 있다.
* 타악기 루프는 원래 반복 구조이므로 반복 확장이 더 자연스럽다.

반복 확장된 샘플은 검색용으로만 사용하고, 실제 반환은 반복 전 원본 샘플로 한다.

## 4초 이상 샘플

4초 이상 샘플은 onset 기준으로 자르되, 최대한 5초 이내의 segment로 맞추는 방향을 논의했다.

다만 하나의 5초 구간으로만 자르면 정보 손실이 클 수 있다.
그래서 긴 샘플은 여러 개의 segment를 만들고, 각 segment별로 MERT embedding을 저장하는 것이 더 안전하다고 봤다.

예:

```text
12초 샘플
→ segment A
→ segment B
→ segment C
```

검색은 segment 기준으로 하고, 반환은 원본 sample 기준으로 한다.

즉:

```text
검색 기준 = 정규화된 반복/절단 segment
반환 기준 = 원본 샘플
```

자른 샘플의 경우도 실제 반환은 자르기 전 원본 샘플이다.
가능하다면 가장 유사했던 구간의 시간 정보도 함께 보여주는 것이 납득 가능성을 높인다.

예:

```text
추천 샘플: 장구_loop_013.wav
가장 유사한 구간: 04.2s ~ 08.9s
유사도: 0.87
```

# MERT 입력 품질

MERT에 넣기 전 음원 품질과 포맷도 논의했다.

MERT-v1 계열은 24kHz 기준으로 알려져 있으므로, 검색용 입력은 다음 조건으로 통일하는 것이 적절하다고 봤다.

```text
sample rate: 24,000 Hz
channel: mono
format: WAV 권장
bit depth: 16-bit PCM 이상
loudness: normalize
silence: 앞뒤 무음 제거
```

원본은 44.1kHz나 48kHz로 보관해도 되지만, MERT 임베딩 추출 직전에는 24kHz mono로 변환한다.

# MERT 임베딩 추출 전처리

논의된 전처리 순서는 다음과 같다.

```text
원본 음원 업로드
↓
파일 포맷 검증
↓
mono 변환
↓
24kHz resampling
↓
앞뒤 무음 제거
↓
볼륨 정규화
↓
짧은 루프 반복 확장
↓
긴 루프 segment 처리
↓
MERT 입력
↓
embedding 추출
↓
vector DB 저장 / 검색
```

세부 내용은 다음과 같다.

## 1. 파일 포맷 검증

허용 포맷 후보:

```text
WAV, FLAC, MP3, M4A
```

검색 품질 관점에서는 WAV/FLAC이 가장 좋고, 저비트레이트 MP3는 피하는 것이 좋다고 봤다.

## 2. mono 변환

스테레오를 mono로 변환한다.

이유:

* 샘플마다 좌우 배치가 다르면 임베딩이 흔들릴 수 있다.
* 타악기 유사 검색에서는 공간감보다 리듬·질감이 중요하다.
* DB와 query 조건을 통일해야 한다.

## 3. 24kHz resampling

MERT-v1 입력에 맞춰 24kHz로 변환한다.

## 4. 앞뒤 무음 제거

leading/trailing silence를 제거한다.

단, 중간의 쉼은 리듬의 일부일 수 있으므로 제거하지 않는다.

## 5. 볼륨 정규화

MVP에서는 peak normalize만 해도 충분하다고 봤다.
더 정교하게는 loudness normalize도 가능하다.

## 6. 짧은 루프 반복 확장

4초 미만은 반복해서 4초 이상으로 맞춘다.

## 7. 긴 루프 처리

4초 이상은 onset 기준으로 segment를 만들고, 각 segment를 최대한 5초 이내로 맞추는 방향을 논의했다.

---

# 검색 기준

최종 검색 기준은 다음과 같이 정리됐다.

```text
길이를 정규화한 샘플들에 대해
업로드한 음원과 MERT embedding cosine similarity가 가까운 것을 검색한다.
```

DB에는 원본 샘플 자체가 아니라 검색용으로 정규화된 segment들의 embedding이 저장된다.

각 embedding은 원본 sample_id와 연결된다.

검색 결과는 segment similarity 기준으로 계산하지만, 사용자에게는 원본 샘플을 반환한다.

---

# 점수 집계 방식

긴 샘플에서 여러 segment가 나오는 경우, 원본 sample 단위의 점수를 어떻게 만들지 논의했다.

단순하게 가장 비슷한 segment만 쓰면 우연히 한 구간만 비슷한 샘플이 올라올 수 있다.

그래서 다음 방식이 제안됐다.

```text
sample_score =
  0.8 * best_segment_similarity
+ 0.2 * top3_segment_similarity_average
```

또는 더 단순한 MVP에서는 best segment similarity만으로도 가능하다고 봤다.

하지만 사람이 듣기에 더 납득 가능한 검색 결과를 위해서는 best segment와 top-k 평균을 섞는 것이 안정적이다.

---

# 반환 방식

반환은 반드시 원본 샘플 기준으로 한다.

정리하면:

```text
4초 미만 샘플:
검색용 = 반복 확장본
반환용 = 반복 전 원본

4초 이상 샘플:
검색용 = onset 기준 segment
반환용 = 자르기 전 원본
추가 표시 = 가장 유사했던 segment 구간
```

이렇게 해야 사용자는 실제 사용할 수 있는 원본 샘플을 받으면서도, 왜 추천됐는지 이해할 수 있다.

---

# 현재 가장 유력한 MVP 구조

현재까지의 논의를 종합하면 가장 유력한 MVP는 다음 구조다.

```text
1. DB 국악 타악기 루프 수집
2. 모든 음원을 mono, 24kHz로 변환
3. 앞뒤 무음 제거
4. 볼륨 정규화
5. 4초 미만 루프는 반복 확장
6. 4초 이상 루프는 onset 기준으로 5초 이하 segment 생성
7. 각 검색용 segment에서 MERT embedding 추출
8. embedding을 vector DB에 저장
9. 사용자가 레퍼런스 샘플 업로드
10. 동일한 전처리 수행
11. query embedding과 DB embedding cosine similarity 비교
12. segment similarity를 sample_id 단위로 집계
13. Top-K 원본 샘플 반환
14. 가능하면 가장 유사했던 구간과 유사도 표시
```

---

# 향후 고도화 방향으로 논의된 것

MVP 이후에는 MERT-only의 약점을 보완할 수 있다.

가능한 고도화 요소:

* BPM 유사도 보정
* onset density 유사도 보정
* onset pattern 유사도 보정
* 악기군 태그 보정
* 장단 metadata 반영
* 장단 cycle 기반 segment
* “킥 대체”, “레이어링”, “텍스처” 같은 용도 기반 필터
* 검색 결과 추천 이유 표시

MVP에서는 MERT-only가 충분히 가능하지만, 최종 제품에서는 “MERT + 리듬 보정”이 더 설득력 있다고 봤다.

---

# 핵심 결론

현재 서비스의 핵심은 다음과 같다.

> 프로듀서가 업로드한 레퍼런스 루프와 유사한 국악 타악기 루프를 찾아주는 MERT 기반 오디오 유사 검색 서비스.

기술적으로는:

> 짧은 루프는 반복 확장하고, 긴 루프는 onset 기준 segment로 나눈 뒤, MERT embedding 유사도로 검색하고 원본 샘플을 반환한다.

기획적으로는:

> “어울리는 샘플 추천”이 아니라 “리듬·어택·질감이 유사한 전통 타악기 샘플 검색”으로 정의하는 것이 MVP에 적절하다.
